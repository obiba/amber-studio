# Static Import Analysis and Conversion Strategy

## Executive Summary

This document provides a comprehensive analysis of all dynamic imports in the Amber Studio application and outlines the strategy for converting them to static imports as part of the Quasar-to-Vite migration. The conversion is necessary because Vite's code-splitting differs from Webpack, and manual chunking strategies must replace automatic lazy loading.

**Total Dynamic Imports Found:** 68 occurrences
- **Router Lazy Loading:** 27 route components
- **Async Components:** 24 defineAsyncComponent calls
- **Quasar Boot Files:** 14 boot file imports (Quasar-generated)
- **Runtime i18n:** 3 dynamic language imports

**Current Bundle Size:** 4.6MB JS + 416KB CSS = ~6.0MB total

## 3. Conversion Strategy

### 3.1 Overall Approach

**Phase 1: Convert to Static Imports**
- Remove all `() => import()` from routes
- Convert all `defineAsyncComponent` to regular imports
- Boot files remain dynamic (handled by Vite plugin)

**Phase 2: Implement Manual Chunking**
- Use Vite's `manualChunks` configuration
- Group related components logically
- Optimize bundle sizes based on usage patterns

**Phase 3: Testing & Optimization**
- Measure bundle sizes before/after
- Test lazy loading behavior
- Optimize chunk strategy based on metrics

### 3.2 Router Conversion Pattern

**Before (Webpack):**
```javascript
const routes = [
  {
    path: '/',
    component: () => import('layouts/MainLayout.vue'),
    children: [
      {
        path: '',
        component: () => import('pages/Dashboard.vue')
      }
    ]
  }
]
```

**After (Vite):**
```javascript
import MainLayout from 'layouts/MainLayout.vue'
import Dashboard from 'pages/Dashboard.vue'
// ... all other page imports

const routes = [
  {
    path: '/',
    component: MainLayout,
    children: [
      {
        path: '',
        component: Dashboard
      }
    ]
  }
]
```

### 3.3 Async Component Conversion Pattern

**Before (Webpack):**
```javascript
import { defineComponent, defineAsyncComponent } from 'vue'

export default defineComponent({
  components: {
    DashboardCounts: defineAsyncComponent(() => 
      import('components/dashboard/DashboardCounts')
    )
  }
})
```

**After (Vite):**
```javascript
import { defineComponent } from 'vue'
import DashboardCounts from 'components/dashboard/DashboardCounts.vue'

export default defineComponent({
  components: {
    DashboardCounts
  }
})
```

### 3.4 i18n Runtime Import Handling

**Current Pattern:**
```javascript
import('quasar/lang/' + langIso)
```

**Strategy:**
1. Pre-import all supported languages statically
2. Create language map for runtime lookup
3. No dynamic imports needed

**Implementation:**
```javascript
import langEn from 'quasar/lang/en-US'
import langFr from 'quasar/lang/fr'
import langEs from 'quasar/lang/es'

const langMap = {
  'en-US': langEn,
  'fr': langFr,
  'es': langEs
}

// Then use: langMap[langIso] instead of import()
```

## 5. Migration Impact Assessment

### 5.1 Bundle Size Impact

**Expected Changes:**
- **Initial Bundle:** May increase by 10-20% (more eager loading)
- **Vendor Bundle:** Should decrease with tree-shaking improvements
- **Total Size:** Target 3-4MB (13-30% reduction)
- **Chunk Count:** Reduce from 40+ to 10-15 logical chunks

### 5.2 Performance Impact

**Positive:**
- Faster page transitions (no lazy load delay)
- Better tree-shaking with static analysis
- Improved caching strategy
- Reduced network requests

**Negative:**
- Larger initial download (mitigated by chunking)
- Less granular code splitting
- All route components loaded upfront per chunk

**Mitigation:**
- Aggressive manual chunking by feature
- Route-based code splitting at chunk level
- Service worker caching strategy
- Compression (gzip/brotli)

### 5.3 Risk Assessment

| Risk | Severity | Mitigation |
|------|----------|------------|
| Bundle size increase | Medium | Manual chunking, tree-shaking optimization |
| Performance regression | Low | Proper chunk strategy, testing |
| Breaking changes | Low | Comprehensive testing of all routes |
| Developer experience | Low | Clear conversion patterns, tooling |
| Build time increase | Low | Vite is faster than Webpack |

## 8. Rollback Plan

If issues arise:

1. **Immediate Rollback:** Revert commits, stay on Quasar/Webpack
2. **Partial Rollback:** Keep static imports, disable manual chunking
3. **Gradual Migration:** Convert routes first, components later

**Rollback Triggers:**
- Bundle size increases > 50%
- Performance degradation > 20%
- Critical bugs in production
- Build process failures
